# 4. Jobs Scraping
 
The application needs job listings from multiple boards to populate the searchable index in `pgvector`. The two boards of primary interest — LinkedIn and Indeed — do not offer open job-search APIs for this use case: LinkedIn provides no public job-search endpoint (access is partner-gated, posting-oriented, and currently closed to new partnerships), and Indeed deprecated its Publisher Jobs API, leaving only employer/ATS-side endpoints. Many other boards, however, do expose clean, sanctioned search APIs. The ingestion strategy therefore combines official APIs where they exist with scraping where they do not.
 
**Decision**: Ingestion is built around a pluggable source interface, with each board implemented as an interchangeable source behind a common abstraction. Official, sanctioned search APIs form the backbone — Adzuna, Jooble, and Careerjet globally, with regional sources (USAJobs, Arbeitnow, the German Bundesagentur feed) added per target market. Apify actors handle the boards without sanctioned APIs (Indeed, optionally LinkedIn) as a supplementary, scraped layer. Apify is chosen over running our own scrapers because it offloads the anti-bot, proxy, and CAPTCHA maintenance. Scraped sources are treated as best-effort and individually switch-off-able; when an API source and a scraped source cover the same listing, the API source wins. Ingestion runs as asynchronous workers via the [queue tool](001-queue-tool.md).
 
**Tradeoff**: Scraped sources carry ToS exposure and can be blocked or break on a markup change — mitigated by the switch-off-able design and the official-API backbone, which keeps the application functional even when scraping fails. Apify bills per run/result: negligible at learning-project scale, a real cost at scale. Combining heterogeneous sources requires a normalization layer (schema mapping and cross-source deduplication), since each source returns a different shape. Relying on scraping for Indeed and LinkedIn means that coverage is less reliable than the API backbone — an accepted tradeoff, given there is no sanctioned alternative for those boards.
 
**Alternatives**:
 
- *DIY scraping (Playwright/Puppeteer)* — full control and no per-run vendor cost, but we would own the entire anti-bot fight: residential proxies, fingerprint handling, CAPTCHA solving, and constant breakage maintenance. Rejected — too much ongoing maintenance for the value when Apify already solves it as a managed service.
- *Aggregator APIs that scrape underneath (Mantiks, Fantastic.jobs)* — present a clean API and claim LinkedIn/Indeed coverage, but the data is scraped under the hood, carrying the same ToS exposure with less transparency and behind a subscription. Useful as a fallback, but Apify gives more direct control over the scraping configuration. Not chosen as primary.
- *Official APIs only (no scraping)* — the cleanest option legally, with stable sources that never break on a markup change. Rejected as the sole strategy because it sacrifices Indeed and LinkedIn coverage entirely; the scraped layer is added on top to recover some of that coverage, kept optional so ingestion degrades gracefully to API-only when scraping is disabled.
